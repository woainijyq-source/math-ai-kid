# AI 儿童脑力互动项目·技术路线决策建议（Web 原型 vs iOS App）

## 1. 目标
本文件用于解决当前项目的一个关键路线问题：

**这个产品是先做 Web，还是直接按 iOS App 来做？**

结合当前已知条件：
- 主设备是 iPad 12.9 寸
- 后续很可能发展为正式 iOS / iPadOS App
- 后面还需要回到 Mac 使用
- 当前又需要尽快开始，用 Claude Code 推进开发

本建议的核心结论是：

> **短期：用 Web 原型快速验证核心体验。**  
> **中期：在验证有效后，转向 iOS / iPadOS 正式版架构。**

也就是说，当前最优路线不是二选一，而是：

## 推荐路线：**Web 原型验证 + iOS 正式版演进**

---

## 2. 为什么不建议一开始就死锁在纯 Web 路线

如果这个产品未来主要在 iPad 上长期使用，那么纯 Web 会有几个明显问题：

### 2.1 原生体验上限有限
这个项目后面很可能会越来越依赖：
- 语音输入与对话
- 摄像头与图像理解
- 动画反馈
- 更沉浸的全屏体验
- 更稳定的本地权限与媒体能力

这些能力在 Web 里都能做，但体验上限、稳定性和交互丝滑度通常不如原生 App。

### 2.2 儿童产品对“顺手感”要求更高
成年人能容忍网页感，小朋友通常不会。  
如果打开、授权、切换、等待、录音这些动作稍微别扭，沉浸感就会断。

### 2.3 后续如果做大，Web 结构可能变成迁移包袱
如果一开始把业务逻辑、状态、内容、AI 调用都紧紧绑在 Web 页面结构里，后面转 iOS 会很痛。

---

## 3. 为什么也不建议现在直接全力原生 iOS 起步

### 3.1 当前最大风险不是“技术做不做得到”，而是“核心体验是否成立”
这个项目还处于产品探索期。  
现在真正要验证的是：
- AI 对手型是否真的最上头
- AI 共创型是否真的更有长期价值
- 奖励系统的“世界变化感”是否真的有效
- 7 岁孩子到底会被哪种 AI 关系吸引

这些问题，原生和 Web 都能验证。

### 3.2 现在最需要的是试错速度
如果一开始就重投入 SwiftUI 原生工程，会显著提高：
- UI 开发成本
- 交互迭代成本
- Demo 修改成本
- Claude Code 的执行复杂度

### 3.3 当前阶段更适合先验证“玩法”，再决定“壳”
这个项目现在的关键，不是“App 长什么样”，而是：

> **什么样的 AI 互动能让孩子愿意天天回来。**

---

## 4. 推荐路线拆解

## 阶段 A：Web 原型验证期
目标：
- 快速做出可玩原型
- 尽快验证产品感觉
- 用最小成本淘汰错误方向

建议做法：
- 用 Web 技术快速做原型
- 先在 iPad 上跑通体验
- 原型重点放在：
  - AI 对手型
  - AI 共创型
  - AI 剧情型

### 这个阶段的目标不是：
- 做最终 UI
- 做最终技术架构
- 做 App Store 级产品

### 这个阶段的目标是：
- 找到最值得长期投入的核心体验

---

## 阶段 B：技术路线决策 Gate
在完成 3 个核心原型试玩后，做一次正式决策。

### 决策问题
1. 当前最强原型是哪一个？
2. 它是否强依赖：
   - 语音
   - 摄像头
   - 高质量动画
   - 本地媒体能力
3. Web 是否已经足够好用？
4. 是否已经值得投入正式版？

### 决策结果二选一
#### 方案 1：继续 Web 做 MVP
适合：
- 原型阶段效果很好，且 Web 体验已足够顺手
- 想先继续低成本扩展和试验

#### 方案 2：转入 iOS / iPadOS 正式版
适合：
- 已经找到很强的核心体验
- 产品未来明确主要跑在 iPad 上
- 对语音、摄像头、沉浸感要求持续提升

---

## 阶段 C：iOS / iPadOS 正式版
如果阶段 B 决定转原生，则建议把正式版目标设为：
- iPad 为主战场
- iPhone 可后续兼容
- Mac 为后续同生态延展

### 当前建议
如果走原生，优先考虑：
- **SwiftUI**（如果明确 Apple 生态优先）

备选：
- React Native / Expo（如果更强调跨平台连续性）

---

## 5. 当前最推荐的技术策略

我给出一个非常明确的建议：

## 建议策略：
### 原型期：Web
### 正式版：SwiftUI 优先评估

原因：
- Claude Code 更适合快速搭 Web 原型
- Web 非常适合现在验证玩法
- 一旦玩法成立，SwiftUI 更适合做长期 iPad 体验
- 这个产品很可能需要高品质原生交互，而不是长期停留在网页感上

---

## 6. 现在开发时必须遵守的“防未来返工”原则

为了保证未来可以平滑迁移到 iOS，当前 Web 原型阶段必须提前做约束。

### 6.1 内容配置与页面分离
不要把任务逻辑写死在页面里。  
要把这些独立出来：
- 任务配置
- 角色定义
- 奖励配置
- 剧情分支
- prompt 模板

### 6.2 AI 服务层独立
不要让页面直接拼 prompt。  
要把 AI 能力封装成统一服务层：
- chat
- stt
- vision
- summary

### 6.3 状态模型独立于页面结构
不要让状态只围绕某个页面组件设计。  
应先定义：
- 用户进度
- 当前任务状态
- 奖励状态
- 世界状态
- 家长摘要状态

### 6.4 原型 UI 可以快，但信息结构要稳
页面样式可以先轻量，但：
- 页面职责
- 模块边界
- 数据流方向
必须清楚。

### 6.5 不要过早深绑定 Web 特性
比如：
- 不要把所有交互都写成网页临时 hack
- 不要把核心能力依赖某个难迁移的浏览器技巧
- 不要把业务逻辑散落在 JSX 中

---

## 7. 文档层面需要同步更新的地方

当前已有文档应统一改成下面这个口径：

### 正确表述
- 当前阶段优先做 **iPad First 的原型验证系统**
- 第一阶段可采用 **Web 原型** 提高迭代速度
- 技术架构需兼顾后续向 **iOS / iPadOS App** 演进
- Mac 为后续兼容终端，而非当前主导约束

### 需要避免的表述
- “本项目就是 Web App”
- “正式形态默认是 PWA”
- “Mac 是与 iPad 并列主战场”

## 8. 给 Claude Code 的当前开发口径

可以这样要求 Claude Code：

> 请按“原型优先、未来可迁移”的原则开发。
> 当前先用 Web 技术快速实现原型，但请把内容配置、AI 服务层、状态模型、奖励系统设计为可迁移结构，不要把业务逻辑死绑在 Web 页面里。

这句话很关键。

---

## 9. 结论

### 最终建议
这个项目现在**不用马上改成纯 iOS 开发**，但也**不能再把自己写死成长期 Web 项目**。

最合理的路线是：

### 现在：
- 用 Claude Code 做 Web 原型
- 优先在 iPad 上验证体验

### 接下来：
- 原型验证后设一个技术路线决策 Gate
- 若体验成立，优先评估 SwiftUI 正式版

### 一句话总结
> **原型阶段追求速度，正式阶段追求体验。**
> **先用 Web 找到灵魂，再用 iOS 做出身体。**

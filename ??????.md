# AI 儿童脑力互动项目·技术选型建议

## 1. 目标
本文件用于给 Claude Code 提供第一版工程实现的技术方向，优先满足：
- iPad 12.9 寸体验优先
- Mac 可兼容
- 快速原型验证
- 便于接入 AI、语音、图片、摄像头
- 后续可平滑扩展为正式产品

---

## 2. 总体建议

### 推荐结论
第一阶段优先采用：
- **前端：Next.js + React + TypeScript**
- **UI：Tailwind CSS + shadcn/ui（可选）**
- **状态管理：Zustand**
- **动画：Framer Motion**
- **音频/语音：Web Audio API + 浏览器录音 + AI 语音识别接口**
- **后端：Next.js API Routes / Route Handlers**
- **数据库：SQLite（原型期）或 Supabase（若需远程同步）**
- **AI 接口：先走统一 server-side AI gateway 封装**
- **部署：本地优先开发，后续可上 Vercel/自托管**

---

## 3. 为什么这样选

### 3.1 为什么当前阶段先做 Web 原型，而不是直接原生 App
原因：
1. 原型迭代速度快
2. Claude Code 对 Web 项目生成和维护效率最高
3. 足够验证 AI 互动、页面结构和核心玩法
4. iPad 和 Mac 都能直接跑，便于当前试错
5. 但正式版不应因此被锁死，后续需保留转向 iOS / iPadOS 的空间

### 3.2 为什么选 Next.js
- 适合前后端一体快速搭建
- 页面和 API 能统一管理
- 对 Claude Code 友好
- 后期从 Demo 升级到正式产品成本较低

### 3.3 为什么选 TypeScript
- 后续会有大量状态、任务配置、角色系统、AI 返回结构
- TS 有助于避免后期结构混乱

### 3.4 为什么选 Zustand
- 比 Redux 轻
- 比 Context 更适合任务状态、奖励状态、对话状态管理
- 原型阶段开发效率高

---

## 4. 推荐技术栈

## 4.1 前端
### 核心
- Next.js
- React
- TypeScript

### 样式
- Tailwind CSS

### UI 组件
可选：
- shadcn/ui

用途：
- 对话卡片
- 按钮
- 面板
- 弹窗
- 表单控件

### 动画
- Framer Motion

用途：
- 页面切换
- 角色反馈
- 奖励解锁动效
- 世界变化动效

---

## 4.2 音频与语音

### 第一阶段建议
#### 输入
- 浏览器麦克风录音
- 录音后上传服务端
- 服务端调用语音识别

#### 输出
两种方案：
1. 先用文字 + 简单音效
2. 进阶再加 TTS 角色语音

### 原因
第一版重点先验证“AI 互动逻辑”，不是先卷语音表现层。

### 后续可加
- 流式语音识别
- 流式 TTS
- 角色音色
- 打断式对话

---

## 4.3 图片与摄像头

### 第一阶段建议
- 使用浏览器原生 `getUserMedia`
- 支持拍照或上传图像
- 不做重实时识别
- 先走“图像上传后交给 AI 解析”的方式

### 为什么这样做
- 开发更稳
- 原型期足够验证玩法
- 避免前期陷入视觉工程泥潭

---

## 4.4 后端

### 推荐
- Next.js Route Handlers / API Routes

负责：
- AI 请求中转
- 任务编排
- 用户数据读写
- 奖励状态更新
- 家长摘要生成

### 原因
- 一体化方便
- 适合单人/小团队快速推进

---

## 4.5 数据存储

### 原型期推荐
#### 方案 A：SQLite
适合：
- 本地原型开发
- 单用户测试
- 快速验证

#### 方案 B：Supabase
适合：
- 需要跨设备同步
- 需要云端数据
- 后续考虑远程访问

### 当前建议
- **先 SQLite**
- 如果后面确认要跨 iPad/Mac 自动同步，再升级到 Supabase

---

## 4.6 AI 层

### 核心原则
不要让前端直接散着调用不同 AI 能力，应该封成统一 AI 服务层。

### 建议分层
#### ai/chat
- 角色对话
- 剧情推进
- 追问和反馈生成

#### ai/stt
- 语音转文本

#### ai/vision
- 图片理解
- 摄像头上传图理解

#### ai/summary
- 家长摘要
- 成长记录

### 好处
- 未来换模型方便
- 可统一 prompt 管理
- 便于调试和记录

- `prompts/`：角色、剧情、追问策略 prompt
- `content/`：任务配置、剧情配置、奖励配置
- `store/`：前端状态管理
- `types/`：任务、角色、奖励、用户画像等类型

---

## 6. 第一阶段不建议过早引入的东西

先不要急着上：
- 重型游戏引擎
- 原生 iOS 开发
- 复杂多人系统
- 实时 3D
- 太复杂的后端微服务拆分
- 过早的多模型编排

原因：
- 现在核心风险不是性能，而是核心体验是否成立

---

## 1. 这份文档回答什么问题

1. **当前这个 MVP 到底已经验证了什么？**
2. **哪些能力只是代码上有基础、但还没真机验证？**
3. **未来如果要做复杂公文 / 制式文书 / 报告的一键排版，WPS 宿主和 AI 各自该负责什么？**
4. **为什么当前 MVP 只做"段落结构分类 + 样式套用"，而不是一上来做全量复杂排版？**

结论先说：

- 当前 MVP 的核心不是"万能排版"，而是先把最小闭环做实：
  **读取段落 → 识别段落结构 → 预览目标样式 → 真实写回样式**。
- 这是后续一切复杂能力的地基。没有这条闭环，表格、页眉页脚、图片环绕、目录刷新都只是概念。
- AI 在本项目里应长期收敛为：**理解结构、输出意图、给出安全建议**；
  WPS 宿主 / API 才是：**读对象、改样式、改版面、改字段、落地执行**。

## 2. 当前产品分层

当前项目建议按 4 层理解：

### L1：宿主接入层（WPS Add-in 宿主）
负责让插件在 WPS 中真正出现并可交互。

当前已验证：
- `规范排版助手` Ribbon Tab 可见
- TaskPane 可打开
- 本地文件模式可作为当前主路线运行

这层解决的是：**插件能不能进到文档现场**。

### L2：文档读取层（WPS JS API 读能力）
负责从当前文档里读取对象状态。

当前已部分打通：
- 可读取段落集合
- 可读取段落文本
- 可读取段落现有样式名
- TaskPane 已提供"读取文档段落"按钮用于真机核验

这层解决的是：**插件是否真的看得见文档内容**。

### L3：结构判断层（AI / 本地启发式）
负责把"段落文本"判断成结构标签，而不是直接改文。

当前已打通：
- Worker 模式：远程分类
- 本地降级：`localHeuristicClassify()`
- 结果必须是与原段落一一对应的结构标签数组

这层解决的是：**怎么看懂文档结构**。

### L4：格式执行层（样式写回）
负责把结构标签映射为 WPS 实际样式和格式动作。

当前已部分打通：
- 可将 `heading1/2/3/body` 映射为 `Heading 1/2/3/正文`
- 已兼容中英样式名候选
- 已提供"预览样式应用"与"真实写回"两条路径

这层解决的是：**怎么把理解结果安全落地**。

## 3. AI 与 WPS 宿主/API 的职责边界

这是后续设计最关键的边界，必须长期坚持。

## 3.1 AI 负责什么

AI 负责"理解"，不负责"直接动文档对象"。

AI 适合负责：
- 段落结构分类
  - 标题 1 / 标题 2 / 标题 3 / 正文
  - 引言 / 注释 / 落款 / 附件说明（未来）
- 版式意图判断
  - 某段是否像主标题、二级标题、图题、表题、页下注
- 复杂文书规则建议
  - 某文书更像通知、报告、纪要、制度、合同、汇报材料
- 风险提示
  - 发现标题层级跳级
  - 发现正文前有疑似空标题
  - 发现表题/图题格式不一致
- 生成"待执行计划"
  - 告诉宿主：哪些段落应改为哪些样式，哪些表格应套什么规则，哪些字段应刷新

AI 不应直接负责：
- 直接回写正文文本
- 直接决定某个 WPS 对象的底层 API 调用细节
- 绕过宿主做图片定位、表格边框、页边距修改
- 在返回结果中夹带"替换后的整段新文本"让前端直接覆盖正文

一句话：
**AI 输出的是结构化决策；不是最终文档本体。**

## 3.2 WPS 宿主 / API 负责什么

WPS 宿主负责"执行"，不负责"理解语义"。

WPS 宿主 / API 适合负责：
- 获取当前文档对象模型
- 遍历段落、表格、图片、节、页眉页脚、域/目录
- 查询对象当前格式状态
- 应用样式、段落格式、页边距、分页、表格样式、图片布局
- 刷新目录、更新域、插入分页符/分节符
- 记录执行结果和失败原因

一句话：
**WPS API 是机械臂；AI 是识别与决策层。**

## 3.3 推荐的长期接口形态

未来不管扩到多复杂，建议都保持这个接口思想：

```json
{
  "documentType": "report",
  "operations": [
    {
      "kind": "paragraph-style",
      "target": { "sourceIndex": 3 },
      "style": "Heading 1",
      "reason": "主标题"
    },
    {
      "kind": "table-normalize",
      "target": { "tableIndex": 2 },
      "preset": "official-report-table"
    }
  ],
  "warnings": [
    "检测到标题层级可能跳级",
    "检测到目录可能需要刷新"
  ]
}
```

也就是说，AI 返回的是**操作建议清单**；真正执行这些操作的是 WPS 宿主。

## 4. 能力地图：按对象层拆解

下面把未来"复杂文书一键排版"拆成 6 个对象层，每层分为：
- **已验证能力**：真机或代码链路已较明确
- **待验证能力**：现有架构可接，但还需要真实 WPS 验证
- **潜在能力边界**：理论上可做，但风险大、成本高或需要更深宿主能力

---

## 4.1 段落层（Paragraph Layer）

### 已验证能力
- 读取段落集合（真机已验证：Paragraphs 对象可访问，Count=5）
- 读取段落文本
- 读取段落当前样式名
- 过滤空段落 / 归一化文本
- 基于 AI 或本地启发式，输出段落结构标签
- 将结构标签映射为目标样式：`Heading 1/2/3/正文`
- 预览 `beforeStyle -> afterStyle`
- 真实写回段落样式

### 待验证能力
- 真机环境下更稳定地兼容不同 WPS JS API 对象形态
- 更多样式别名兼容（如不同中文包、定制模板样式）
- 标题层级跳级检测
- 列表项 / 编号项 / 项目符号项识别
- 段前段后距、行距、首行缩进、对齐方式等段落格式统一
- 正文首段、章节首段、附录段等特殊段落策略

### 潜在能力边界
- 直接基于正文语义推断完整公文逻辑层级，误判风险较高
- 同一段中包含多种格式片段（run 级别）时，简单套段落样式不够
- 对"看起来像标题但其实是正文句首强调"的误判，需要更强规则约束

### 判断
段落层是当前 MVP 的核心，也是复杂排版里**ROI 最高、风险最低、最值得优先做实**的一层。

---

## 4.2 样式层（Style Layer）

### 已验证能力
- 已建立结构标签到目标样式名的基本映射
- 已考虑中英样式候选名
- 已具备"预览后执行"的安全路径
- 样式对象可访问（真机已验证：Styles 对象可访问，Count=478）

### 待验证能力
- 枚举当前文档可用样式并回显
- 根据模板自动选择最匹配样式
- 区分"内置样式"和"当前单位模板自定义样式"
- 样式不存在时的兜底策略
  - 创建样式
  - 回退到 Normal / 正文
  - 提示用户手工选择映射
- 一份文书内的样式一致性检查

### 潜在能力边界
- 不同模板中同名样式可能定义不同，光靠名称不够
- 某些单位模板依赖手工维护的复杂样式体系，纯自动映射可能不稳定
- 如果未来支持"自动创建样式"，要非常注意不要污染用户现有模板

### 判断
样式层是"段落层能力"走向"可交付排版结果"的关键中间层。当前虽已起步，但离"复杂文书稳定交付"还有明显距离。

---

## 4.3 表格层（Table Layer）

### 已验证能力
- **Tables 对象可访问**（真机已验证：2026-03-16）
- 从架构上，未来完全可以在 TaskPane 路线上新增"表格读取 / 表格预览 / 表格规范化"按钮或操作流

### 待验证能力
- 枚举文档中的表格对象（获取表格数量、遍历表格）
- 获取行列数、单元格文本、表头行位置
- 识别"数据表 / 签批表 / 落款表 / 空白表单"等常见表格类型
- 套表格样式、统一边框、统一对齐方式、统一行高列宽
- 对表题与表格主体联动处理

### 潜在能力边界
- 表格是复杂文书自动化中的高风险区：
  - 合并单元格
  - 嵌套表格
  - 跨页表格
  - 手工画线表格
  - 表格内混排段落与图片
- "看起来像表格"的版式可能并不是真表格，而是制表位 / 文本框 / 多列排版模拟
- 不同单位模板对表格规范差异非常大，不适合过早做成"一键自动强写"

### 判断
表格层适合放在段落层稳定之后，优先做"读取 + 识别 + 预览"，不要一上来就全面自动写回。

---

## 4.4 图片层（Image / Shape Layer）

### 已验证能力
- **Shapes 对象可访问**（真机已验证：2026-03-16）
- **InlineShapes 对象可访问**（真机已验证：2026-03-16）

### 待验证能力
- 读取图片 / Shape / InlineShape 对象（获取数量、遍历对象）
- 识别图片标题（图题）与图片锚点附近段落
- 统一图片对齐方式、环绕方式、尺寸上限
- 图题样式统一
- 图片与图题、正文间距统一

### 潜在能力边界
- WPS 中图片可能分为浮动对象、嵌入对象、形状对象，API 形态不一定统一
- 复杂文书里大量"示意图 + 文本框 + 箭头 + 组合对象"，自动处理成本很高
- 图文混排很容易破版，必须坚持"先预览、再执行、可跳过"的原则

### 判断
图片层不是 MVP 第一优先级。它对"最终观感"影响大，但对"最小可用闭环"不是必须条件。

---

## 4.5 页面层（Page / Section Layer）

### 已验证能力
- **Sections 对象可访问**（真机已验证：2026-03-16，Count=1）
- **Headers 对象可访问**（真机已验证：2026-03-16，Count=3）
- **Footers 对象可访问**（真机已验证：2026-03-16，Count=3）
- **PageSetup 对象可访问**（真机已验证：2026-03-16，TopMargin=72, BottomMargin=72）

### 待验证能力
- 统一页边距、纸张大小、页眉页脚距离（修改 PageSetup 属性）
- 统一节的页面设置（操作 Sections 对象）
- 插入分页符 / 分节符
- 首页不同、奇偶页不同等常见页面规则控制
- 文书首页版式、签发页、附件页的页面策略

### 潜在能力边界
- 页面是结果层，背后往往依赖段落、表格、图片共同作用
- 某些布局需求不能只靠页面设置解决，可能涉及节拆分、对象锚点重排
- 一旦处理不当，最容易出现"明明结构对了，但分页全乱了"的高感知事故

### 判断
页面层应在段落/样式层稳定、表格与图片层具备基本可控性之后再强化，否则很容易进入"修一个地方坏一片"的状态。

---

## 4.6 目录 / 字段层（TOC / Field Layer）

### 已验证能力
- **Fields 对象可访问**（真机已验证：2026-03-16）
- 从产品方向上，这是段落样式稳定后的自然延伸层

### 待验证能力
- 根据标题样式生成 / 刷新目录（操作 Fields 对象）
- 刷新页码、引用、交叉引用、目录域（更新 Fields）
- 识别并保留已有目录区域
- 执行排版后提示"需要刷新目录/字段"

### 潜在能力边界
- 目录 / 字段通常依赖 WPS 对 Word-like 域对象的支持深度
- 目录错误有时不是字段问题，而是前面的标题样式不规范
- 如果未来支持自动刷新目录，需注意不要误覆盖用户手工编辑的伪目录

### 判断
目录 / 字段层非常适合做成"排版完成后的第二阶段动作"，不建议在当前 MVP 中一次性揉进主流程。

## 5. 为什么当前 MVP 只覆盖"段落结构分类"

这不是能力退让，而是故意聚焦。

## 5.1 因为这是最小可验证闭环

当前 MVP 已经具备并正在验证的链路是：

1. 插件进入 WPS 宿主
2. TaskPane 可见
3. 读取段落
4. 判断结构
5. 预览目标样式
6. 真实写回样式

只要这 6 步稳定，项目就已经从"概念 demo"升级为"可持续迭代的真实产品底座"。

## 5.2 因为段落结构是复杂排版的上游

复杂文书中的很多后续能力，本质上都依赖段落结构先对：
- 目录生成依赖标题层级
- 页面分页常常依赖标题和正文位置
- 图题表题识别，本质上也依赖段落上下文
- 许多模板规则都是"某类段落应该长什么样"

如果连段落结构都不稳，后面所有高级能力都会建立在沙地上。

## 5.3 因为段落样式写回的风险最可控

相比表格、图片、页面：
- 段落样式写回更可预测
- 可预览、可对比、可回归排查
- 对正文内容本身侵入最小
- 出错时更容易限制影响范围

## 5.4 因为 AI 在这里最容易被约束住

让 AI 回答"这段更像标题还是正文"，比让 AI 直接控制整份文档版式安全得多。

当前 MVP 刻意把 AI 收敛到：
- 输入：段落文本
- 输出：结构标签
- 不允许：改写文本、直接排版、直接碰宿主对象

这条边界非常对，应该继续坚持。

## 6. 推荐 roadmap（分阶段演进）

下面给一个更贴当前代码现实的路线图。

## Phase 0：宿主接入收口（已基本完成）

目标：先证明插件在真实 WPS 里能活。

已完成/基本完成：
- Ribbon 可见
- TaskPane 可开
- 本地文件模式建立
- TaskPane 四按钮流程固定

成功标志：
- 不再纠结"插件到底有没有加载"
- 排查主战场转移到 TaskPane 内功能链

## Phase 1：段落读取 + 结构分类 MVP（当前主战场）

目标：把"段落读取 → 结构分类 → 预览 → 写回"做稳。

范围：
- 读取前 N 段文本与样式
- Worker 连接测试
- 未配置 Worker 时本地启发式降级
- 标题/正文四分类
- 样式预览
- 样式真实写回
- 真机留档与失败归因

成功标志：
- 一份乱格式文档可在 10 秒级内完成基础结构化排版
- 能明确区分：宿主问题、配置问题、分类问题、写回问题

## Phase 2：样式层增强 + 模板适配

目标：让 MVP 从"能跑"变成"更像真实交付"。

建议加入：
- 枚举当前文档可用样式
- 样式映射配置面板
- 标题层级跳级检测
- 段落格式细化（行距、缩进、对齐、段前后）
- 单位模板 / 场景模板切换
- "预览差异"显示更丰富

成功标志：
- 不同模板下的稳定性显著提升
- 不是只会套 `Heading 1/2/3/正文` 名称，而是真能适配常见模板环境

## Phase 3：表格 / 图题 / 表题的结构化扩展

目标：从纯段落，扩到复杂文书中最常见的非段落对象。

建议加入：
- 表格对象读取与分类
- 图题 / 表题识别
- 表格规范化预览
- 图片与图题联动规范化
- "只预览不执行"的对象级别安全开关

成功标志：
- 插件开始具备处理"正文 + 表格 + 图片标题"的复合文书能力

## Phase 4：页面 / 目录 / 字段联动

目标：进入真正的"整份文档收尾自动化"。

建议加入：
- 页面设置统一
- 分页 / 分节策略
- 自动刷新目录与字段
- 排版完成后的一致性检查
- 生成执行报告（哪些改了，哪些跳过了，哪些需要人工复核）

成功标志：
- 能处理"结构 + 页面 + 目录"三位一体的文档收尾

## Phase 5：复杂文书一键排版

目标：接近最初愿景，但仍然坚持可解释、可预览、可跳过。

建议形态：
- 先识别文书类型
- 再生成执行计划
- 再按层执行：段落 → 样式 → 表格 → 图片 → 页面 → 目录/字段
- 全流程输出审计日志
- 用户可选择"全自动 / 半自动 / 仅预览"

成功标志：
- 不再只是"给段落套标题样式"
- 而是能对一份复杂文书输出**有边界、有解释、有回滚空间**的规范化结果

## 7. 对当前代码路线的建议

基于现状，建议继续坚持下面几条，不要回退：

## 7.1 固定"本地文件模式 + TaskPane 四按钮"

原因：
- 当前真实问题已经进入功能链，而不是加载链
- 四按钮天然适合作为分阶段验收入口
- 后续表格 / 页面 / 目录能力，也都能沿这个 TaskPane 模式继续长出来

## 7.2 保持"先预览、再执行"的产品原则

尤其是未来扩到表格、图片、页面后，这条原则会越来越重要。

建议长期保留：
- 读取/检测按钮
- 预览按钮
- 真执行按钮
- 清晰日志与失败归因

## 7.3 AI 返回结果要继续结构化，不要回到自由文本

当前 `formatter.js` 里的边界是对的：
- 校验段落数量一致
- 校验 style 枚举合法
- 非法就中止

这套思路未来扩到表格、图片、页面时也应保持：
**宁可拒绝执行，也不要模糊执行。**

## 7.4 文档层能力扩展要遵循"先读后写、先段落后对象、先预览后自动"

推荐顺序：
1. 段落
2. 样式
3. 表格
4. 图片
5. 页面
6. 目录/字段

这个顺序比"同时全做"更稳，也更容易留档和定位问题。

## 8. 当前阶段的成功定义

如果要判断这个项目当前阶段是否成功，不应该问：
> "它能不能一次性搞定所有复杂公文排版？"

而应该问：
> "它是否已经把最小闭环做成了一个可迭代的宿主内产品？"

当前更合理的成功定义是：
- WPS 宿主接入稳定
- TaskPane 四按钮口径稳定
- 段落读取可验证
- Worker / 本地启发式双路径可验证
- 样式预览与写回可验证
- 真机结果有留档，有失败归因

只要这几项站稳，后面的复杂文书能力就不是"重做一个项目"，而是在现有骨架上继续分层加能力。

## 9. 一句话总结

**当前 MVP 不是"只会四个按钮的小 demo"，而是在故意把问题收敛到最关键的一层：段落结构识别与样式写回。**

只要这一层在 WPS 真机里做稳，后面的表格、图片、页面、目录/字段，才有机会从"想法"变成"可控扩展"。
